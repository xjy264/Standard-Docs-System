# AGENTS.md

## 适用范围
- 本文件适用于仓库根目录及其所有子目录。
- 本仓库为“大同房建公寓段标准化资料管理系统”，目标是构建可内网和公网部署的企业级文件资料管理系统。
- 开发规范以“真实可演示、稳定可迭代、最小改动、中文业务文案准确”为目标。

## 沟通约定
- 默认使用中文；用户明确要求英文时再切换。
- 开始实现前，先确认需求、约束、改动范围、验证方式和主要风险。
- 涉及前端 UI 或交互改动时，先说明页面结构、入口、状态来源和接口依赖。
- 如果需求、现有代码、文档三者不一致，先暂停并向用户确认。
- 所有输出不得使用 emoji。

## 强制约束
- 使用最小 diff。
- 修改必须可自动应用。
- 不修改无关代码、无关接口、无关页面。
- 不随意改动数据库表结构，除非任务明确要求。
- 不改动已有中文注释和中文业务文案，除非存在明确错别字或需求要求修改。
- 所有中文业务文案必须原样保留，除非用户明确要求调整。
- 修改后前后端必须保持可运行。
- 修改后必须说明改了哪些文件、为什么改、如何验证。

## 仓库结构
- `backend/`: Java 21 + Spring Boot 3 + Spring Security + MyBatis-Plus 后端。
- `frontend/`: Vue 3 + Vite + TypeScript + Element Plus 前端。
- `deploy/`: Docker Compose、Nginx、MySQL 初始化脚本和部署配置。
- `docs/`: 需求、架构、数据库、接口、部署、演示计划等文档。
- `README.md`: 项目说明、启动方式和演示账号。

## 开发原则

### 1. Worktree First
- 只要 Codex 需要改动代码，优先从 `main` 新建独立工作树和 `codex/` 前缀分支。
- 纯文档查看、只读排查、纯本地验证可留在当前工作区。
- 如果在工作树中发现用户已有改动，不得擅自覆盖或回滚，必须先说明情况。

### 2. 文档先行
- 非 trivial 改动先更新 `docs/` 中的需求、接口、数据库或部署文档。
- 影响接口、表结构、权限模型、部署方式时，必须同步更新文档。
- README 必须保留启动方式、演示账号、部署依赖和验证步骤。

### 3. 测试与验证
- 对可自动化的后端逻辑改动，优先补充单元测试或接口测试。
- 前端改动至少执行构建验证。
- 部署相关改动至少说明 Docker Compose 验证方式。
- 禁止“只改代码，不说明怎么验证”。

### 4. 数据库变更
- MySQL 只保存文件元数据，不保存真实文件二进制内容。
- 真实文件默认存储到 MinIO，本地存储只作为开发或降级方案。
- 涉及数据库结构变更时，优先追加初始化或迁移脚本，不覆盖历史脚本。
- 涉及高风险 `UPDATE`、`DELETE`、批量修复数据时，必须说明目的、影响范围和回滚思路。

## 后端开发约定
- 使用 Java 21、Spring Boot 3、Spring Security、JWT、MyBatis-Plus、MySQL 8、Redis、MinIO、Knife4j。
- Controller 只放请求编排、认证信息获取和参数校验；业务逻辑放到 service。
- 所有接口统一返回结构：
  - `code`: 业务状态码。
  - `message`: 中文提示。
  - `data`: 响应数据。
- 必须有统一异常处理。
- 必须封装系统级权限校验，不在接口中散写权限判断。
- 必须封装文件访问权校验，例如 `FileAccessService.canAccess(userId, fileId)`。
- 删除默认软删除，彻底删除必须额外判断权限并记录操作日志。
- 登录、注册审批、文件操作、权限变更、组织变更、系统配置变更必须记录操作日志或登录日志。

## 权限设计约定
- 权限分为两层：系统级权限和文件访问权。
- 系统级权限控制功能入口，例如 `file:upload`、`file:delete`、`user:approve`、`dept:manage`。
- 文件访问权只判断用户能否访问某个文件。
- 第一阶段不拆分文件级别的查看、下载、预览、编辑、抄送权限。
- 用户拥有某文件访问权，就默认可以查看、预览、下载、在线编辑和抄送。
- 删除、恢复、彻底删除仍然需要系统级权限或超级管理员身份。
- 角色模板只作为权限初始化模板，不作为最终权限唯一来源。
- 用户单独权限支持 `allow` 和 `deny`，显式 `deny` 优先。

## 文件管理约定
- 文件可见范围使用 `visibility_scope`：
  - `PRIVATE`: 仅上传人可见。
  - `DEPT`: 上传人所在部门可见。
  - `DEPT_AND_CHILD`: 上传人所在部门及下级部门可见。
  - `ASSIGNED`: 指定人员或指定部门可见。
  - `PUBLIC`: 全段可见。
- CAD 第一阶段只保证上传、识别、下载和未配置预览提示，不强行实现在线预览。
- Office 在线预览和编辑优先预留 OnlyOffice 配置、接口和页面入口；完整回写闭环可分阶段完善。
- 部门抄送采用生成用户接收记录的方式，便于统计已读未读。

## 前端开发约定
- 使用 Vue 3、Vite、TypeScript、Element Plus、Pinia、Vue Router、Axios。
- 页面风格参考中国铁路 12306：铁路政企风格、深蓝主色、白色或浅灰背景、清晰导航、稳重表格和查询区。
- 不做互联网娱乐风格，不使用卡通化设计，不做过度动画。
- 菜单和按钮必须受系统级权限控制。
- 文件列表只展示当前用户有访问权的文件。
- 用户对文件有访问权时显示查看、预览、下载、在线编辑、抄送。
- 删除、恢复、彻底删除按钮必须额外根据系统级权限或超级管理员身份显示。

## 部署约定
- 部署同时兼容内网和公网。
- Docker Compose 至少包含 MySQL、Redis、MinIO、后端、前端、Nginx。
- 通过 `.env` 配置域名或 IP、端口、MinIO 外部访问地址、后端接口地址等。
- 生产密码、密钥、服务器地址不得硬编码到业务代码中。
- 默认初始化管理员账号：`admin`，初始密码：`Admin12345@@`，数据库中必须保存加密后的密码。

## 常用命令

### 后端
- 本地测试：`cd backend && mvn test`
- 本地启动：`cd backend && mvn spring-boot:run`
- 打包：`cd backend && mvn clean package`

### 前端
- 安装依赖：`cd frontend && npm install`
- 本地开发：`cd frontend && npm run dev`
- 生产构建：`cd frontend && npm run build`

### 部署
- 启动：`cd deploy && docker compose up -d`
- 停止：`cd deploy && docker compose down`
- 查看日志：`cd deploy && docker compose logs -f`
