# AGENTS.md

## 适用范围
- 本文件适用于仓库根目录及其所有子目录。
- 本仓库为“大同房建公寓段标准化资料管理系统”，目标是构建可内网和公网部署的企业级文件资料管理系统。
- 开发规范以“真实可演示、稳定可迭代、最小改动、中文业务文案准确、安全脱敏、验证有证据”为目标。

## 权威与优先级
- 本文件是本仓库 agent 协作、开发、验证和交付流程的权威来源。
- 优先级为：用户显式指令 > 本文件 > 当前对话默认行为或历史记忆。
- 如本文件内部条款出现冲突，以更具体、约束更强、风险更低的条款为准；数据库、权限、安全、部署和数据删除相关约束始终不放松。
- 必须优先读取当前仓库真实代码、README 和 `docs/` 文档，不凭印象或其他项目经验直接改动。
- 如果用户需求、现有代码、文档三者不一致，先暂停并向用户说明差异，确认后再继续。

## 沟通约定
- 默认使用中文；用户明确要求英文时再切换。
- 开始实现前，先确认需求、约束、改动范围、验证方式和主要风险。
- 涉及前端 UI 或交互改动时，先说明页面结构、入口、状态来源和接口依赖。
- 涉及 `AGENTS.md` 修改时，需说明借鉴来源、拟调整位置、影响范围和验证方式；如果用户已明确要求直接修改，可在限定范围内执行并在完成后逐条说明差异。
- 所有输出不得使用 emoji。

## 强制约束
- 使用最小 diff。
- 修改必须可自动应用。
- 不修改无关代码、无关接口、无关页面、无关文档。
- 不随意改动数据库表结构，除非任务明确要求。
- 不改动已有中文注释和中文业务文案，除非存在明确错别字或需求要求修改。
- 所有中文业务文案必须原样保留，除非用户明确要求调整。
- 修改后前后端必须保持可运行。
- 修改后必须说明改了哪些文件、为什么改、如何验证。
- 不输出、提交或记录密码、JWT、Cookie、MinIO 密钥、数据库密码、服务器真实密钥等敏感信息；验证时只报告是否存在、长度或脱敏摘要。
- 不主动加入第一阶段未要求的 AI 对话、知识库、短信、邮箱、企业微信或钉钉能力。
- 不使用破坏性 Git 操作覆盖用户改动；遇到脏工作区、冲突、权限问题或非预期改动必须先报告。

## 仓库结构
- `backend/`: Java 21 + Spring Boot 3.3.x + Spring Security + MyBatis-Plus + Flyway + MySQL 8 + Redis + MinIO + Knife4j 后端，使用 Maven 构建。
- `frontend/`: Vue 3 + Vite + TypeScript + Element Plus + Pinia + Vue Router + Axios 前端。
- `deploy/`: Docker Compose、Nginx、MySQL 初始化脚本和部署配置。
- `backend/src/main/resources/db/migration/`: Flyway 迁移脚本。
- `docs/`: 需求、架构、数据库、接口、部署、演示计划等文档。
- `run.sh`: 本地部署、迁移、备份、恢复、故障导出和自检入口。
- `README.md`: 项目说明、启动方式、演示账号、部署依赖和验证步骤。

## 开发原则

### 1. 目标驱动与简单优先
- 将用户任务先改写为可验证目标，再实施。
- 实现前显式写出关键假设；如果自己不确定，不得假装确定。
- 若需求存在多种合理解释，先列出分歧与取舍，不能静默选择其一。
- 优先采用能解决当前问题的最小方案，不为单次需求引入未被要求的通用层、抽象或未来扩展。
- 只清理由本次改动直接造成的废弃 import、变量、函数或注释；已有无关问题只报告，不擅自删除。

### 2. Worktree First
- 只要 Codex 需要修改仓库跟踪内容，默认从本地 `main` 新建独立工作树和 `codex/` 前缀分支。
- 纯文档查看、只读排查、纯本地验证可留在当前工作区。
- 如果当前路径已经是 `git worktree`，或已经在非 `main` 的任务分支上，可直接在当前分支继续，不要重复创建嵌套 worktree。
- 用户明确要求只微调 `AGENTS.md`、手工测试记录或其他不影响运行时的说明文档时，可按用户指示在当前分支处理；若当前工作区需要保护，仍优先使用 worktree。
- 新 worktree 默认放在 `.worktrees/<task-name>`，分支命名使用 `codex/<task-name>`。
- `.worktrees/` 必须保持被 Git 忽略；创建前如发现未忽略，先暂停说明。
- worktree 分支默认不推送远端，除非用户明确要求推送该任务分支。
- 开发任务完成并验证通过后，默认将本次 `codex/` 子分支提交并合并回本地 `main`；如遇冲突、未提交改动、远程权限问题或用户另有要求，必须先暂停并说明情况。
- 如果在工作树中发现用户已有改动，不得擅自覆盖或回滚，必须先说明情况。

### 3. 文档先行
- 非 trivial 改动先更新 `docs/` 中的需求、接口、数据库或部署文档。
- 影响接口、表结构、权限模型、部署方式、登录方式、文件预览方式时，必须同步更新文档。
- 仅调整 agent 协作规则时，修改 `AGENTS.md` 即可，不需要额外制造 `docs/` 变更。
- README 必须保留启动方式、默认演示账号、部署依赖和验证步骤。

### 4. 测试与验证
- 对可自动化的后端逻辑改动，优先补充单元测试或接口测试。
- 前端改动至少执行 `cd frontend && npm run build`。
- 后端改动至少执行 `cd backend && mvn test`，除非明确说明环境原因无法执行。
- 数据库迁移或部署相关改动至少说明 `./run.sh migrate`、`./run.sh doctor` 或 Docker Compose 验证方式。
- 纯文档或 `AGENTS.md` 改动至少执行 `git diff --check`，并人工核对修改范围。
- 禁止“只改代码，不说明怎么验证”；完成说明必须包含实际执行过的验证命令和结果。

### 5. 数据库变更
- MySQL 只保存文件、附件、用户、组织、权限、日志和通知等元数据，不保存真实文件二进制内容。
- 真实文件默认存储到 MinIO，本地存储只作为开发或降级方案。
- 新数据库结构变更优先追加 Flyway 迁移脚本；部署初始化脚本需要兼容时，再同步更新 `deploy/mysql-init/`。
- 不覆盖、重写或删除已执行过的历史迁移脚本。
- 涉及高风险 `UPDATE`、`DELETE`、批量修复数据时，必须说明目的、影响范围、备份方式、回滚思路和验证查询。

## 后端开发约定
- 使用 Java 21、Spring Boot 3.3.x、Spring Security、HttpOnly Cookie 登录态、CSRF 防护、MyBatis-Plus、Flyway、MySQL 8、Redis、MinIO、Knife4j。
- Controller 只放请求编排、认证信息获取和参数校验；业务逻辑放到 service。
- 所有接口统一返回结构：
  - `code`: 业务状态码。
  - `message`: 中文提示。
  - `data`: 响应数据。
- 必须有统一异常处理；普通用户错误提示不得暴露 HTTP 状态码、程序异常、端口号、堆栈或底层服务名称。
- 系统级权限校验必须封装在 `PermissionService` 或业务 service 中，不在接口里散写权限判断。
- 资料目录、文件入口、附件下载、预览、提交记录和回收站访问必须在后端做业务权限校验，不能只依赖前端隐藏按钮。
- 删除默认软删除，彻底删除或自动清理必须额外判断权限、保留必要审计信息，并说明数据影响。
- 登录、注册审批、文件操作、权限变更、组织变更、系统配置变更必须记录操作日志或登录日志。

## 权限与资料业务约定
- 超级管理员不受系统级权限、业务权限、组织归属或显式 `deny` 约束，默认拥有任意权限和任意数据访问/操作能力。
- 权限分为两层：系统级权限和资料业务权限。
- 系统级权限控制功能入口，例如 `user:view`、`user:approve`、`role:view`、`dept:manage`、`log:view`、`storage:view`。
- 资料业务权限控制科室资料目录、文件入口、上传任务、下达文件、车间提交记录、附件下载/预览和回收站。
- 角色模板只作为权限初始化模板，不作为最终权限唯一来源。
- 用户单独权限支持 `allow` 和 `deny`，显式 `deny` 优先。
- 前端菜单和按钮必须按系统级权限控制；后端仍必须做最终权限校验。
- 删除、恢复、彻底删除、系统错误查看、存储统计和权限配置按钮必须额外根据系统级权限或超级管理员身份显示。

## 文件与资料管理约定
- 当前资料目录以 `sys_doc_node` 表示多级目录节点，节点类型包含文件夹和文件。
- `sys_doc_item` 表示文件入口，按 `business_type` 区分上传任务 `UPLOAD` 和下达文件 `ISSUED`。
- 文件可见范围当前使用 `visibility_scope`：
  - `ALL`: 全部车间可见。
  - `SELECTED`: 仅 `visibleWorkshopIds` 指定车间可见。
- 科室资料页统一展示上传任务和下达文件，不恢复旧的“上传 / 下达”双 tab。
- 文件年份优先由父级年度目录继承，前端文件弹窗不单独展示年份选择。
- 上传任务可配置单份提交或多份提交，并可配置多个收集项；所有收集项都上传附件后才视为一次完整提交。
- 下达文件通过富文本内容和下达附件承载公示资料，不参与上传任务进度统计。
- 科室文件删除后进入本科室共享回收站；恢复时必须选择同科室目标目录。
- PDF 和图片直接内联预览；Office 文件通过 OnlyOffice 预览服务和短时附件访问票据拉取，不在 URL 中暴露登录 JWT。
- CAD 第一阶段只保证上传、识别、下载和未配置预览提示，不强行实现在线预览。

## 前端开发约定
- 使用 Vue 3、Vite、TypeScript、Element Plus、Pinia、Vue Router、Axios。
- 页面风格参考中国铁路 12306：铁路政企风格、深蓝主色、白色或浅灰背景、清晰导航、稳重表格和查询区。
- 不做互联网娱乐风格，不使用卡通化设计，不做过度动画。
- 菜单和按钮必须受系统级权限控制。
- 文件列表只展示当前用户有业务访问权的文件。
- 用户对文件有访问权时显示查看、预览、下载、在线编辑预留入口和抄送/提交相关入口；具体按钮仍以当前业务状态和权限为准。
- 删除、恢复、彻底删除按钮必须额外根据系统级权限、资料业务权限或超级管理员身份显示。
- 前端错误提示应面向普通用户温和可理解；详细诊断进入系统错误上报和后端日志，不直接展示堆栈。

## 部署与运维约定
- 部署同时兼容内网和公网。
- Docker Compose 至少包含 MySQL、Redis、MinIO、后端、前端、Nginx；OnlyOffice 作为 Office 预览依赖按需启用。
- 通过 `deploy/.env` 或环境变量配置域名或 IP、端口、MinIO 外部访问地址、后端接口地址、JWT 密钥和 OnlyOffice 地址。
- 生产密码、密钥、服务器地址不得硬编码到业务代码中。
- 默认演示管理员登录手机号：`00000000000`，初始密码：`Admin12345@@`，数据库中必须保存加密后的密码。
- 系统不提供跳过密码的管理员免密登录入口。
- 内网故障导出通过 `./run.sh export-errors --days <N>` 或超级管理员“个人空间 / 我的提醒 / 系统错误”页签执行，导出内容必须脱敏。

## 常用命令

### 后端
- 本地测试：`cd backend && mvn test`
- 本地启动：`cd backend && mvn spring-boot:run`
- 打包：`cd backend && mvn clean package`

### 前端
- 安装依赖：`cd frontend && npm install`
- 本地开发：`cd frontend && npm run dev`
- 生产构建：`cd frontend && npm run build`

### 部署
- 初始化生产环境配置：`./run.sh init-prod-env`
- 启动：`./run.sh deploy`
- 迁移：`./run.sh migrate`
- 自检：`./run.sh doctor`
- 停止：`cd deploy && docker compose down`
- 查看日志：`cd deploy && docker compose logs -f`
